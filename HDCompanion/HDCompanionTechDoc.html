1. Design Pattern & Deployment Structure

The application follows a decoupled, modular Model-View-ViewModel (MVVM) inspired architecture built entirely on native Windows PowerShell 5.1. The interface is defined purely in strict XAML (Extensible Application Markup Language), while the logic is segmented into specialized PowerShell Modules (`.psm1`).

To facilitate ease of deployment, the application utilizes a Flat Directory Structure. All core engine files, UI templates, and logic modules exist in a single root folder. This prevents relative pathing errors when distributed via SCCM or network shares.

Module Responsibilities

  • Main.ps1: The Entry Point. Initializes state, imports dependencies, loads the XAML, and enters the WPF Dispatcher loop.
  • CoreLogic.psm1: Handles Global State, centralized JSON configuration binding, custom XAML parsing, API calls, and Audit Logging.
  • UIEvents.psm1: The Controller. Binds PowerShell script blocks to XAML control events (Clicks, TextChanged) and manages thread contexts.
  • RemoteManagement.psm1: The WMI/CIM/WinRM execution engine.

2. Security & Execution Context

Helpdesk Companion does not use stored credentials or run as a service account. It executes exclusively under the Active Directory token of the user who launches `Main.ps1`.

  • Least Privilege: If a standard user launches the application, AD queries will work (read-only), but password resets, unlocks, and Remote Management actions will fail gracefully via standard Windows Access Denied exceptions.
  • Execution Policy: The launch shortcut utilizes powershell.exe -ExecutionPolicy Bypass to execute the scripts without altering the host machine's global execution registry keys.
  • Audit Trail: The application injects $env:USERNAME and $env:COMPUTERNAME into the centralized CSV logs for every critical action (Unlocks, Resets, Uninstalls) to maintain compliance tracking.

3. WPF & Strict XAML Parsing

The application features a custom UI loader (Load-XamlWindow in CoreLogic.psm1) that dynamically injects theme colors and strictly parses XML to comply with modern .NET Framework security policies.

DTD Processing Security Fix

To prevent XamlParseException errors when spawning dynamic inline windows (like the Run Task or Message popups), the custom parser explicitly overrides the default .NET XML security restrictions to allow local DTD processing:

$xmlSettings = New-Object System.Xml.XmlReaderSettings
$xmlSettings.DtdProcessing = [System.Xml.DtdProcessing]::Parse
$reader = [System.Xml.XmlReader]::Create([System.IO.StringReader]::new($xamlText), $xmlSettings)
return [System.Windows.Markup.XamlReader]::Load($reader)

4. Remote Management Engine (WinRM)

The RemoteManagement.psm1 module powers the System Manager. Due to modern firewalls blocking legacy RPC (Remote Procedure Call) and DCOM ports, the application was refactored to tunnel all endpoint administration exclusively through WinRM (Windows Remote Management) via Invoke-Command.

The Deserialization Challenge

Because WinRM serializes objects into XML to transmit them over the network, standard objects returned from endpoints (like Processes or Profiles) arrive as PSObject deserialized data. The WPF `GridView` control famously fails to bind to these properties.

The Solution: UIEvents.psm1 strictly rebuilds every single array returned by the Remote Management module into fresh [PSCustomObject] wrappers before assigning them to the UI's ItemsSource.

Silent Software Uninstallation Logic

The Uninstall-RemoteSoftware function uses an intelligent parser to execute background uninstalls. It queries the remote registry for the QuietUninstallString. If unavailable, it analyzes the standard UninstallString to determine the installer framework:

  • MSI (msiexec): Replaces /I with /X and appends /qn /norestart.
  • InnoSetup (unins000.exe): Appends /VERYSILENT /SUPPRESSMSGBOXES.
  • Nullsoft (uninstall.exe): Appends /S.

The resulting string is then executed natively on the remote host via WMI's Win32_Process Create method inside a WinRM block.

5. Active Directory Optimization

The ActiveDirectory.psm1 module relies on the RSAT Active Directory PowerShell module. To ensure the interface does not lock up during the 30-second background polling cycle, AD queries use highly optimized server-side filtering.

Lockout Querying

Search-ADAccount -LockedOut -UsersOnly -ResultPageSize 2000

We use `Search-ADAccount` instead of `Get-ADUser -Filter {LockedOut -eq $true}` because the former executes directly against the Domain Controller's pre-computed bitmask index, making it exponentially faster in large environments.

PDC Event Log Parsing (Lockout Source)

The "Find Lockout Source" feature does not query local domain controllers. It dynamically discovers the Forest's PDC Emulator (Primary Domain Controller) and queries its security log for Event ID 4740. The PDC is the authoritative source where all bad password attempts are ultimately forwarded.

6. Freshservice API V2 Implementation

The application integrates with the Freshservice ITSM to provide hardware asset lookups based on AD data. It authenticates via Basic Auth utilizing a Base64 encoded API token provided via hdcompanioncfg.json.

User-to-Computer Lookup Pipeline

Because assets in Freshservice are assigned to "Requesters", resolving an AD User to a Computer requires a chained two-step API call:

  1. Requester Query: The app takes the user's AD EmailAddress property and passes it to the Requesters endpoint using V2 Query syntax: /api/v2/requesters?query="primary_email:'user@domain.com'" to retrieve the internal integer user_id.
  2. Asset Query: The app then queries the Assets endpoint using that ID: /api/v2/assets?query="user_id:{id}" to retrieve the array of assigned hardware.

Error Handling: The API module includes an exception interceptor that extracts the hidden JSON response body from HTTP 400 Bad Request errors to expose the specific Freshservice syntax failure to the UI.
